Footprint Cluster Chart

1Summary

A footprint (cluster) chart is a price-level order-flow grid. For every candle it shows, at each price, how much volume traded at the bid (sell-initiated) versus at the ask (buy-initiated). It answers a question standard OHLC candles cannot: who was aggressive, and at which price, inside the bar.

This is a headline pro-trader feature on Dhan DEXT, GoCharting, NinjaTrader, and Bookmap, and a natural fit for Nubra's active F&O / scalper audience. This document scopes v1, describes the working simulation that accompanies it, specifies what is required from frontend and backend, and presents our findings from live API testing, the v3 docs, and backend confirmation. The headline finding: the raw fields needed appear to already exist in the tick-level orderbook websocket (last traded price + quantity per tick, with synchronized depth in the same packet). The open item is confirming the stream's tick granularity, not building a feed from scratch.

2Feature scope & expectations

A grid, not a chart: columns are candles along a time x-axis, rows are price levels on a shared y-axis, and each cell holds two numbers, bid volume (left) and ask volume (right), for that price in that candle. We deliberately exclude candlestick bodies, VWAP, volume profile, and a cumulative-delta pane; those make it a chart, and are out of scope for v1.

What we expect it to do

  • Show per-price-level bid vs ask traded volume for each candle, on a shared price axis so a row is the same price across all candles.
  • Stream the current candle live; freeze closed candles permanently; backfill recent history on open.
  • Surface the analytical reads a footprint is used for: imbalance, per-candle delta, and the highest-volume level.

3What the simulation demonstrates

The Live Simulation tab is a working proof-of-concept, not a mockup. It runs the real algorithm we would ship, the quote-rule classifier plus price-level and per-candle aggregation, on a synthetic trade tape. Only the raw ticks (price and quantity per trade) are fabricated; every displayed number is computed by the actual pipeline.

It therefore validates two things at once: the frontend behaviour (shared-axis grid, live current candle with frozen history, imbalance, POC, delta footer, hover, timeframe/symbol switching, index disabled-state, bidirectional scroll) and the exact data shape we need from backend. If the backend can supply that shape, the frontend is de-risked.

5Backend requirements (data contract)

The frontend needs one primitive: each executed trade's price + quantity, which it classifies and buckets itself. This likely already streams; the contract below is what we consume.

ItemRequirement
SourceThe tick-level orderbook websocket: per tick, ltp (last traded price), ltq (last traded quantity), volume, and the top bid/ask levels, in one packet.
Granularity (to confirm)A message per executed trade, so ltq is each individual print's size (not coalesced across trades). This is the one open verification.
ClassificationDone client-side via the quote rule (Lee & Ready 1991): trade at/above ask = buy, at/below bid = sell, else tick-rule fallback. Uses the bid/ask in the same packet, so no separate quote feed needed.
CoverageAll instruments with an order book: cash equity + F&O. Indices excluded (no book).
AggregationClient-side: bucket classified volume by price level (configurable step) and by candle (1m / 3m / 5m / 15m).

6Findings: current data availability

Investigated via the v3 /charts/timeseries endpoint (authenticated PROD calls), the v3 REST + WebSocket docs, and backend confirmation.

DataStatusDetail
cumulative_volume_delta Available (F&O) Live-tested at 1-second interval: a NIFTY ATM option returned a real, per-second, cumulative delta series (14,776 points, all non-zero, ~14,775 distinct, rising 171 → 4.95M through the session). This is the CVD line. RELIANCE (cash stock) was 0 at every one of 19,930 one-second points, confirming it is computed for F&O only. It is one cumulative value per timestamp, not a per-price-level breakdown.
l1bid / l1ask Available Returned as real series down to 1-second interval in the same test (for both the option and the stock). Confirms top-of-book is captured at high frequency, which is one half of what quote-rule classification needs.
orderbook websocket
(ltp, ltq, volume + 20-level depth)		 Available The realtime orderbook channel carries last traded price, last traded quantity (ltq), volume, and up to 20 bid/ask levels per message. The docs state that with no interval set, the default is tick-level streaming. This is the raw ingredient a trade tape is built from, delivered with the synchronized quote in the same packet.
Dedicated trade tape / footprint buckets
(pre-classified by aggressor)		 Compute-side No endpoint returns per-trade data pre-tagged buy/sell-initiated, or ready-made per-level footprint buckets. These are computed client-side from the tick-level orderbook stream: capture ltp+ltq per tick and classify against the same packet's bid/ask via the quote rule.

Interpretation. Two separate conclusions. (1) CVD and per-bar delta are available today for F&O: cumulative_volume_delta is a real cumulative delta series down to 1-second resolution (confirmed live), so the CVD line is a direct plot and the Delta Strip is its per-bar difference. For cash equities the field is 0, so those need building. (2) The footprint grid is a different requirement: it needs per-price-level buy/sell volume, which this scalar field does not provide. That per-level data is reconstructed from the tick-level orderbook stream (per-tick last price + last quantity + synchronized depth); the one open check is whether a tick fires per executed trade (so ltq is each print's size) or coalesces trades. Note the existence of a real F&O cumulative_volume_delta proves the backend already classifies executed trades by aggressor internally, so per-level buckets and cash coverage are extensions of an existing pipeline, not a greenfield build.

7The ask to backend

Request

Already available (no ask): CVD and per-bar delta for F&O. cumulative_volume_delta is live down to 1-second resolution (confirmed). Three items remain:

  • Footprint per-level data. Confirm the tick granularity of the default (tick-level) orderbook stream: does a message fire per executed trade with ltq = that single trade's quantity, or coalesce trades? If per-trade, we build the per-level footprint client-side (capture ltp+ltq per tick, classify against the same packet's bid/ask). If coalesced, expose a true tick-by-tick trade-event feed.
  • Cash-equity delta/CVD. cumulative_volume_delta returns 0 for cash stocks. Extend the existing F&O aggressor-classification pipeline to cash equities.
  • Leverage what exists. A real F&O cumulative_volume_delta proves aggressor classification is already done server-side. Per-level footprint buckets and cash coverage are extensions of that pipeline, not greenfield.
Have: tick-level orderbook (ltp + ltq + depth)→ Quote-rule classify (same packet's bid/ask)→ Aggregate by level + candle→ Footprint

This rides existing infrastructure (the l1 quote + depth streaming already in place) rather than a greenfield build.

Where the four features stand: CVD (3) and Delta Strip (2) are available for F&O today. If the tick-level stream is per-trade, that also unlocks the Footprint grid (1) and Large Trade Bubbles (4), plus delta/CVD for cash, on the existing stream, with no new feed.

8Open questions & dependencies

  • On the tick-level orderbook stream, is a message emitted per executed trade (ltq = that print's quantity), or coalesced per order-book change?
  • Can the existing F&O cumulative_volume_delta pipeline be extended to cash equities, and to per-level granularity?
  • Can this be delivered over the existing market websocket, or is a new channel needed?
  • Any per-instrument or brokerage-tier entitlement gating on this data?
  • What historical depth is available for backfill?

9Risks & limitations

  • Quote-rule classification is an approximation, not exchange-tagged aggressor truth. Adequate for v1 (it is what comparable retail tools use); a true aggressor feed is a later accuracy upgrade.
  • The dense grid is desktop-first; mobile needs a separate design pass.
  • Thin/illiquid instruments produce sparse footprints of limited value.
  • A live footprint is data-heavy; it requires the websocket feed, not REST polling.
  • v1 uses same-level imbalance; diagonal imbalance is a v2 refinement.

DOM / Market Depth

1Summary

Two features built on the order book (resting bid/ask liquidity, "market depth"): the DOM Imbalance Meter (total bid vs total ask liquidity as a single gauge) and the DOM Heatmap (how much liquidity sits at each price level across recent snapshots, Bookmap-style).

Both read the order book (resting depth), which Nubra already streams up to 20 levels live. The headline finding: this cluster is largely buildable on existing APIs today. The only new dependency is optional (historical L2 depth, and only for heatmap replay).

The two DOM features: scope closure

Featurev1Data neededDisposition & ask
1. DOM Imbalance Meter In v1 Order-book depth (bid/ask quantity per level), up to 20 levels Available today. Sum bid vs ask quantity from the existing depth feed and render the ratio. No backend ask. Advanced: weight by distance from LTP (each level carries its price).
2. DOM Heatmap (live) In v1 The same depth feed, streamed; recent snapshots accumulated client-side Available today for the live, going-forward heatmap. Stream the orderbook channel and keep a rolling buffer of the last N snapshots. No backend ask.
2b. DOM Heatmap (history / replay) Optional Historical L2 depth snapshots over time The only new dependency in this cluster. No endpoint exposes past depth, so backfill / replay ("what did the book look like an hour ago") needs a new historical L2 feed. Not required for the live heatmap.

3What the simulations demonstrate

Both tabs are working proofs-of-concept running the real logic on a synthetic order book. The Imbalance Meter sums bid vs ask quantity across the live depth and renders the true ratio; the Liquidity Heatmap keeps a rolling buffer of the last five depth snapshots and colours each price level by liquidity, exactly as a production build would from the orderbook stream. Only the raw depth numbers are synthetic.

7The ask to backend

Request (small)

Most of this cluster needs no backend work: the Imbalance Meter and the live Heatmap both run on the existing orderbook depth feed. Two items to close:

  • Confirm 20-level depth in PROD. We have verified 5 levels live; docs say up to 20. Confirm the full 20 levels stream in production for the instruments we care about.
  • Optional: a historical L2 depth feed, only if we want heatmap replay / backfill. Not needed for the live heatmap.

Both features ride the order book, which already exists as a live feed, so the cluster needs little to no backend work to ship.

8Open questions

  • Does the PROD orderbook stream deliver the full 20 levels for all instruments, or fewer for some?
  • Any per-instrument or tier entitlement gating on depth beyond 5 levels?
  • Is a historical L2 depth store feasible if we later want heatmap replay?
  • Websocket throughput on dense books (20 levels x many instruments) for the heatmap.

9Risks & limitations

  • Raw bid/ask ratio is noisy on thin instruments; thresholds must be tuned per instrument to avoid false urgency.
  • The imbalance gauge says how one-sided the book is, not where that liquidity sits relative to price (distance-weighting is the advanced answer).
  • Resting depth can be spoofed / pulled; it is intent, not executed volume, so it should not be read as guaranteed liquidity.
  • The heatmap is render-heavy on mobile; desktop-first.
  • Live-only heatmap has no memory before subscription; replay needs the optional historical feed.
